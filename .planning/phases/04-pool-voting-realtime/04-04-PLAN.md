# Plan 04-04: Supabase Realtime subscription for live vote count updates

## Objective

Add the Supabase Realtime subscription to the Club Detail screen that listens for `pool_books` row update events and triggers a TanStack Query invalidation — so when any member casts or removes a vote, all connected devices see the vote count update within seconds without a manual refresh (VOTE-03 / D-10).

The subscription is scoped to the `pool-{clubId}` channel, active only while the Pool tab is mounted (`activeTab === 'pool'`), and automatically unsubscribed on tab change or component unmount (D-11).

<action>
In `expo-app/app/(app)/clubs/[id]/index.tsx`, add the following import at the top (if not already present):

```typescript
import { useEffect } from 'react';
```

(Note: if `useState` is already imported from `'react'`, add `useEffect` to the same import.)

Then add the following `useEffect` inside the `ClubDetailScreen` component body, after the mutations and before the JSX return statement:

```typescript
// Realtime subscription for live vote count updates (D-10, D-11, VOTE-03)
// Subscribe when pool tab is active; unsubscribe when tab changes or component unmounts.
useEffect(() => {
  if (activeTab !== 'pool' || !id) return;

  const channel = supabase
    .channel(`pool-${id}`)
    .on(
      'postgres_changes',
      {
        event: 'UPDATE',
        schema: 'public',
        table: 'pool_books',
        filter: `club_id=eq.${id}`,
      },
      () => {
        // Any pool_books row update (vote_count changed) → invalidate pool query.
        // TanStack Query refetches in the background; UI updates when data arrives.
        queryClient.invalidateQueries({ queryKey: ['pool_books', id] });
      }
    )
    .subscribe();

  return () => {
    channel.unsubscribe();
  };
}, [id, activeTab, queryClient]);
```

**Notes:**
- The channel name `pool-${id}` matches D-11 exactly.
- The `filter: 'club_id=eq.{id}'` scopes the subscription to this club only — avoids receiving updates from other clubs' pools.
- No `INSERT` or `DELETE` event handling — only `UPDATE` events fire when `vote_count` changes via the RPC. Propose (INSERT to pool_books) and admin delete (DELETE from pool_books) are handled by direct query invalidation in their respective mutations (already in 04-02 and 04-03).
- Do NOT add `event: 'INSERT'` or `event: 'DELETE'` here — those mutations already invalidate the query directly, and adding Realtime for them would cause double-fetches.
- The `activeTab !== 'pool'` guard is critical: when the user switches to Members or Settings tab, the subscription is cleaned up immediately. When they switch back to Pool, a new subscription is created.
</action>
